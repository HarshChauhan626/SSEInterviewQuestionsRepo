00 / overview

Introduction

GraphQL and gRPC are both alternatives to plain REST, built to solve different pain points. GraphQL was created at Facebook (2012, open-sourced 2015) to solve over-fetching and under-fetching of data for client applications with diverse, fast-changing UI needs. gRPC was created at Google (2015, based on their internal "Stubby" system) to solve fast, strongly-typed, low-latency communication between backend services.

They are not really competitors — in most real systems they coexist: gRPC is used for internal service-to-service (east-west) traffic, while GraphQL (or REST) is used for the external, client-facing (north-south) edge. Understanding both deeply, and where each one fits, is exactly what senior engineering interviews probe for.

API Styles Compared: REST vs GraphQL vs gRPC

Before comparing GraphQL and gRPC to each other, it helps to place both against the baseline most engineers already know: REST.

Dimension REST GraphQL gRPC
Transport HTTP/1.1 (or 2) HTTP/1.1 or 2 (usually POST to one endpoint) HTTP/2 only
Payload format JSON (usually) JSON Protocol Buffers (binary)
Schema / contract Optional (OpenAPI) Mandatory, strongly typed (SDL) Mandatory, strongly typed (.proto)
Endpoints Many (resource-based URLs) Single endpoint (/graphql) Many RPC methods, single connection
Over/under-fetching Common problem Solved by design (client picks fields) N/A — request shape is fixed by contract
Streaming No (native) Subscriptions (via WS/SSE) Native: client, server & bidi streaming
Browser support Native Native (it's just HTTP+JSON) Needs grpc-web + proxy
Caching Easy (HTTP caching, CDNs) Hard (single endpoint, POST) Hard (binary, no URL semantics)
Typical use Public APIs, simple CRUD Client-facing aggregation layer (BFF) Internal microservice-to-microservice
Performance Medium Medium (resolver overhead) High (binary + HTTP/2 multiplexing)
Codegen Optional (via OpenAPI) Yes (typed clients from schema) Yes, first-class — stubs in 10+ languages
Mental model
REST = resources over URLs. GraphQL = a query language over a typed graph of data, optimized for flexible client reads. gRPC = typed remote procedure calls over a fast binary wire, optimized for service-to-service speed and contracts.
02 / graphql

GraphQL

A query language for APIs and a runtime for executing those queries against a typed schema, giving clients the power to ask for exactly the data they need — nothing more, nothing less.

What it is, and why it exists

Classic REST APIs expose fixed-shape resources. A mobile screen that needs a user's name, their last 3 orders, and each order's shipping status either has to call multiple endpoints (under-fetching, more round trips) or call one heavy endpoint that returns far more than it needs (over-fetching, wasted bandwidth). GraphQL fixes this by exposing your data as a graph of types and letting the client send a query describing the exact shape of the response it wants.

Problems it solves

  • Over-fetching / under-fetching
  • API versioning sprawl (v1, v2, v3...)
  • Multiple round trips for nested data
  • Frontend teams blocked on backend changes

What it's good at

  • Client-facing APIs with varied consumers (web, iOS, Android)
  • Aggregating multiple backend services (BFF pattern)
  • Rapidly evolving frontends
  • Strongly typed, self-documenting contracts

Schema & Type System (SDL)

Every GraphQL API is described by a Schema Definition Language (SDL) document. This schema is the contract — the single source of truth for what can be queried.

# Scalar & object types
type User {
  id: ID!
  name: String!
  email: String!
  orders(limit: Int = 10): [Order!]!
}

type Order {
  id: ID!
  status: OrderStatus!
  total: Float!
  items: [OrderItem!]!
}

enum OrderStatus { PENDING SHIPPED DELIVERED CANCELLED }

type Query {
  user(id: ID!): User
  searchUsers(query: String!): [User!]!
}

type Mutation {
  createOrder(input: CreateOrderInput!): Order!
}

type Subscription {
  orderStatusChanged(orderId: ID!): Order!
}

Key type-system rules: ! means non-nullable, [Type!]! means a non-null list of non-null items, and every operation root (Query, Mutation, Subscription) is itself just a regular object type. GraphQL ships scalar types Int, Float, String, Boolean, ID — anything else (dates, JSON, etc.) needs a custom scalar.

Interfaces & Unions

GraphQL supports polymorphism so a field can return one of several possible types:

interface Node { id: ID! }
type Article implements Node { id: ID! title: String! }
type Video implements Node { id: ID! durationSec: Int! }

union SearchResult = Article | Video | User

Queries, Mutations & Subscriptions

GraphQL has exactly three operation types — this is one of its core best-practice constraints: reads, writes, and live updates are distinct and explicit.

Operation Purpose Semantics
Query Read data Side-effect free; fields resolve in parallel
Mutation Write / mutate data Top-level fields resolve serially, in order
Subscription Live/streaming updates Long-lived connection (typically WebSocket), event-driven 
// Query: client asks for exactly these fields, nested
query GetUserOrders($userId: ID!) {
  user(id: $userId) {
    name
    orders(limit: 5) {
      id
      status
      total
    }
  }
}

// Mutation: write + select back the result shape
mutation PlaceOrder($input: CreateOrderInput!) {
  createOrder(input: $input) {
    id
    status
  }
}

// Subscription: stays open, pushes events
subscription WatchOrder($orderId: ID!) {
  orderStatusChanged(orderId: $orderId) {
    status
  }
}
Fragments
Reusable field selections: fragment OrderFields on Order { id status total } then spread with ...OrderFields. Keeps large client queries DRY and is heavily used by codegen tools like Apollo Client / Relay.

Resolvers & Execution Model

Every field in the schema is backed by a resolver function — (parent, args, context, info) => value. When a query arrives, the GraphQL engine walks the selection set and calls a resolver per field, passing the parent field's resolved value down. If a field has no custom resolver, a sensible default (read the property off the parent object) is used. 

const resolvers = {
  Query: {
    user: async (_parent, { id }, ctx) => ctx.db.users.findById(id),
  },
  User: {
    orders: async (user, { limit }, ctx) =>
      ctx.db.orders.findByUserId(user.id, { limit }),
  },
};

Execution is a tree walk: GraphQL resolves the root field first (e.g. user), then recursively resolves each child field of the result in parallel (for queries) before moving deeper. This is exactly why naive resolver implementations cause the N+1 problem below.

The N+1 Problem & DataLoader

This is one of the most-asked GraphQL interview topics. If you query 10 users and each user resolver triggers a separate DB query for its orders, you get 1 query for users + N queries for orders — the N+1 problem.

N+1 fan-out vs DataLoader request batching + caching

DataLoader (Facebook's reference implementation, ported to most languages) fixes this with two techniques per request:

  • Batching — collects all .load(id) calls made within a single tick of the event loop and issues one batchLoadFn(ids) call, typically translating to WHERE id IN (...).
  • Caching — memoizes results per request so the same ID is never fetched twice within one GraphQL operation.
const orderLoader = new DataLoader(async (userIds) => {
  const rows = await db.orders.findByUserIds(userIds);
  return userIds.map(id => rows.filter(r => r.userId === id));
});

// resolver becomes a single batched call
User: { orders: (user, _args, ctx) => ctx.orderLoader.load(user.id) }
Interview trap
DataLoader must be instantiated per request, never as a singleton — otherwise its cache leaks data across different users' requests.

Architecture Patterns

GraphQL is most often deployed as a gateway / Backend-for-Frontend (BFF) layer in front of several underlying services — REST APIs, gRPC services, databases.

GraphQL as an API gateway aggregating heterogeneous backends

Common deployment patterns

Pattern Description When to use
Monolithic GraphQL server One server, schema + resolvers + direct DB access Small/medium apps, single team
GraphQL as BFF Gateway aggregates multiple internal services (often gRPC) Microservices with one client-facing API
Schema stitching Merge multiple independent GraphQL schemas at runtime Legacy approach, mostly superseded by federation
Apollo Federation / GraphQL Federation Each service owns a subgraph; a router composes a supergraph Large orgs, many teams owning different domains

Best Practices

  • Design schema around use cases, not your database. The schema is a product surface, not a 1:1 mirror of tables.
  • Always use DataLoader (or equivalent batching) for any field that triggers a DB/network call.
  • Paginate every list field — prefer cursor-based Relay-style connections (edges, node, pageInfo) over offset pagination for stability under concurrent writes.
  • Set query complexity / depth limits to prevent malicious or accidental deeply-nested queries from overwhelming the server.
  • Version by deprecation, not by URL. Use @deprecated(reason: "...") on fields instead of /v2/graphql.
  • Use persisted queries in production to avoid sending full query text over the wire and to allow allow-listing.
  • Return nullable types for fields that can fail so one failing field doesn't null out the entire response (avoid over-using !).
  • Centralize error handling using structured error extensions (code, not just message strings).
  • Cache at the field/resolver level (e.g. with response cache directives) since HTTP caching doesn't work out of the box.
  • Keep resolvers thin — push business logic into services; resolvers should just orchestrate.

Pitfalls & Security

Common pitfalls

  • N+1 queries from un-batched resolvers
  • No pagination → unbounded list responses
  • Schema mirrors DB schema 1:1 (leaky abstraction)
  • Overly permissive introspection in production
  • No request timeout / complexity limit → DoS risk

Security checklist

  • Query depth limiting & cost analysis
  • Disable introspection (or restrict) in production
  • Authorization at the field level, not just at the gateway
  • Rate limit by query cost, not just request count
  • Sanitize/validate all input types server-side
Classic attack: nested query DoS
A single malicious query like user { friends { friends { friends { ... } } } } can be exponential in cost. Mitigate with max query depth, query cost/complexity scoring, and request timeouts.
03 / grpc

gRPC

A high-performance, open-source RPC framework built on HTTP/2 and Protocol Buffers, designed for fast, strongly-typed communication between services — first-class support for streaming, code generation, and pluggable load balancing/auth.

What it is, and why it exists

Where GraphQL optimizes the client-facing read path, gRPC optimizes the internal call path between backend services. Instead of hand-rolling JSON REST clients per language, a team defines a service contract once in a .proto file, then generates strongly-typed client and server stubs in dozens of languages. Calls look like local function calls but execute over the network, using a compact binary wire format instead of text-based JSON.

Problems it solves

  • Slow, verbose JSON serialization between services
  • Manually written, drifting client SDKs per language
  • No native support for streaming RPCs
  • Head-of-line blocking on HTTP/1.1 connections

What it's good at

  • Low-latency service-to-service calls
  • Polyglot microservices (codegen in 10+ languages)
  • Streaming workloads (telemetry, chat, file transfer)
  • Strict contracts with backward-compatible evolution

Protocol Buffers (protobuf)

Protobuf is the interface definition language (IDL) and binary serialization format gRPC uses by default. A .proto file declares messages (data) and services (RPC methods), which the protoc compiler turns into native classes/structs plus client+server stubs.

// user.proto
syntax = "proto3";
package shop.v1;

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  repeated Order recent_orders = 4;
}

message Order {
  string id = 1;
  OrderStatus status = 2;
  double total = 3;
}

enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
  ORDER_STATUS_SHIPPED = 2;
}

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc WatchOrders(WatchOrdersRequest) returns (stream Order);
}

message GetUserRequest { string id = 1; }
message WatchOrdersRequest { string user_id = 1; }

The integer after each field (= 1, = 2...) is the field tag — it, not the field name, is what's actually encoded on the wire. This is the key to protobuf's backward/forward compatibility model.

Schema evolution rules

Safe change Breaking change
Adding a new field with a new tag number Reusing a tag number for a different field
Removing a field (reserve its tag/name) Changing a field's tag number
Renaming a field (tag stays the same) Changing a field's type incompatibly (e.g. string → int)
Adding new enum values Changing the meaning of an existing enum value
Adding a new RPC method Changing an RPC method's request/response type
Interview trap
Always reserve removed field tags/names so a future field can never accidentally reuse them and silently corrupt old clients reading new data: reserved 4, 7; reserved "old_field";

On the wire, protobuf uses varint encoding for small integers and a tag-length-value (or tag-value for varints) layout, which is why payloads are typically 3–10x smaller than the equivalent JSON, with much cheaper parsing.

The 4 RPC Types

This is the single most-tested gRPC concept in interviews — gRPC supports four call shapes, all defined just by where the stream keyword appears in the .proto service definition.

The four gRPC call shapes — all multiplexed over a single HTTP/2 connection

Type Shape Typical use case
Unary 1 request → 1 response Standard CRUD-style calls (GetUser)
Server streaming 1 request → N responses Live feeds, large result sets paged over the wire
Client streaming N requests → 1 response File/chunk upload, batched telemetry ingestion
Bidirectional streaming N requests ↔ N responses Chat, collaborative editing, real-time gaming state

HTTP/2 Internals That Make gRPC Fast

gRPC requires HTTP/2, and most of its performance characteristics come directly from features HTTP/1.1 doesn't have:

  • Multiplexing — many concurrent RPCs share one TCP connection as independent streams, eliminating head-of-line blocking at the connection level and the cost of opening new connections per call.
  • Binary framing — HTTP/2 frames (not text headers) are cheaper to parse than HTTP/1.1's textual request lines.
  • Header compression (HPACK) — repeated headers (auth tokens, content-type) across calls on the same connection are compressed via a shared dynamic table instead of resent in full.
  • Native, long-lived streaming — a single HTTP/2 stream can carry an open-ended sequence of length-prefixed protobuf messages in either direction, which is what powers the four RPC types above.
  • Flow control — per-stream and per-connection windows prevent a fast sender from overwhelming a slow receiver.
Why browsers can't speak gRPC directly
Browsers don't expose low-level HTTP/2 trailer and framing control to JavaScript, so a browser can't originate a true gRPC call. grpc-web solves this with a JS client that talks a constrained gRPC-over-HTTP/1.1-or-2 variant to an Envoy (or similar) proxy, which translates to real gRPC for the backend.

Architecture Patterns

gRPC's natural home is the service mesh / internal microservices layer, often paired with a GraphQL or REST edge for clients and a sidecar proxy (e.g. Envoy) per service for cross-cutting concerns.

gRPC microservices behind sidecars in a service mesh, fronted by an edge gateway

Common deployment patterns

Pattern Description When to use
Direct service-to-service gRPC Services call each other's gRPC stubs directly Small/medium service counts, simple topology
Service mesh (Envoy/Istio/Linkerd) Sidecar proxies handle mTLS, retries, LB, tracing Large microservice fleets needing uniform policy
gRPC gateway / transcoding Reverse proxy exposes gRPC services as REST/JSON for external callers Need both internal gRPC and a public REST API from one service
grpc-web + Envoy Browser clients talk gRPC indirectly via a translating proxy Browser apps that need gRPC's typed contracts

Best Practices

  • Always set deadlines/timeouts on every call and propagate them downstream — without this, a slow dependency cascades into resource exhaustion across the whole call chain.
  • Never reuse or reassign protobuf field tag numbers. Reserve removed ones.
  • Design idempotent unary RPCs where possible so client-side retries are safe; use retry budgets / exponential backoff with jitter.
  • Use streaming only when you need it — unary is simpler to reason about, load balance, and retry than long-lived streams.
  • Version services at the package level (package shop.v1; → shop.v2;) rather than per-message ad hoc versioning.
  • Use interceptors (gRPC's middleware) for cross-cutting concerns: auth, logging, metrics, tracing — keep them out of business logic.
  • Enable mTLS between services in production; never trust the network layer alone for service identity.
  • Keep messages reasonably small; for large payloads (files, blobs) use client/server streaming with chunking rather than one giant message.
  • Use well-known status codes (NOT_FOUND, UNAVAILABLE, DEADLINE_EXCEEDED...) consistently so clients can branch on them instead of parsing error strings.
  • Load test with realistic concurrency — HTTP/2 multiplexing means a single misbehaving stream can still affect others on a shared connection under certain flow-control conditions.

Pitfalls

Common pitfalls

  • No deadlines set → cascading slow-call failures
  • Treating streaming RPCs as a default rather than an exception
  • Load balancing at L4 (TCP) instead of L7 for long-lived HTTP/2 connections, causing uneven traffic distribution
  • Reassigning a protobuf field tag → silent data corruption
  • No browser support assumed without grpc-web + proxy

Operational checklist

  • Client-side and server-side deadlines configured
  • Retries are idempotency-aware, with backoff + jitter
  • mTLS / auth interceptors on every service boundary
  • L7-aware load balancing (e.g. via mesh or client-side LB)
  • Reserved tags/fields documented in every .proto
Interview trap: connection-level load balancing
Because HTTP/2 multiplexes many calls over one long-lived TCP connection, a naive L4 load balancer pins all of a client's calls to a single backend instance once the connection is established — defeating load balancing entirely. Fix with client-side load balancing, an L7-aware proxy, or a service mesh.
03.5 / performance

Performance & Benchmarks

Raw performance numbers vary by workload, but the fundamental characteristics of each protocol determine where overhead lives and how to mitigate it.

Where Each Protocol Spends Its Time

Phase REST GraphQL gRPC
Serialization JSON encode/decode (~medium) JSON encode/decode + query parse (~medium-high) Protobuf encode/decode (~3–10× faster than JSON)
Wire payload size Medium (full resource) Medium (only requested fields) Small (binary, field-tag only, no names on wire)
Connection overhead HTTP/1.1: new TCP+TLS per req (or keep-alive pool) Same as REST One persistent HTTP/2 connection, many streams
Server-side parsing URL routing + JSON decode SDL query parse + validation + resolver plan Stub dispatch + protobuf decode
Caching Easy (GET + URL = cache key) Hard (POST + dynamic body) Hard (binary, no URL semantics)
Latency (p50 internal) ~1–5 ms (JSON) ~2–8 ms (resolver overhead) ~0.3–1 ms (binary+multiplexing)
Benchmark caveat
Micro-benchmarks showing gRPC at 5–10× REST are often measuring serialization in isolation. In real systems, DB/network I/O dominates; gRPC's advantage is most pronounced in high-fanout internal call chains where many small fast calls accumulate.

Real-world Optimization Tactics

GraphQL performance levers

  • DataLoader — eliminates N+1, single biggest win
  • Persisted queries — skip full query parse, enable GET caching
  • Response caching — cache at resolver or whole-operation level
  • Query complexity limits — prevent runaway resolver trees
  • Deferred queries (@defer) — stream critical fields first
  • CDN edge caching — via persisted query GET requests

gRPC performance levers

  • Connection pooling — reuse HTTP/2 connections across calls
  • Keepalive tuning — prevent idle connection teardown
  • Client-side load balancing — spread load before a single conn pins
  • Message size limits — chunk large payloads via streaming
  • Compression — gzip/zstd per call for large text payloads
  • Flow control tuning — adjust HTTP/2 window sizes for throughput

05 / decision guide

Choosing the Right Tool

Scenario Best fit Why
Public API consumed by many third parties REST or GraphQL Browser-native, cacheable, broad tooling support
Mobile app with varied screens / bandwidth constraints GraphQL Client controls exact payload shape, fewer round trips
Internal call between two backend microservices gRPC Binary + HTTP/2, lowest latency, typed contracts, codegen
Real-time bidirectional data (chat, collab editing) gRPC (internally) / GraphQL subscriptions (at edge) gRPC bidi streaming is the most efficient transport; GraphQL exposes it simply to clients
Polyglot microservice fleet (Go, Java, Python, etc.) gRPC First-class codegen keeps every language's client in sync with the contract
Simple CRUD service, single consumer, fast to ship REST Lowest ceremony, no schema tooling required
Aggregating data across many existing services for one app GraphQL (as BFF) Purpose-built for combining heterogeneous sources into one response
Rule of thumb
If the consumer is a browser or a varied set of external clients, lean GraphQL (or REST). If the consumer is another backend service you control, lean gRPC. Most non-trivial systems eventually use both, each at the layer it's strongest in.
07 / interview bank

Interview Q&A — 30 Questions

Organized by topic. Click any question to reveal the answer. All questions are senior-engineer / system-design level.

List 1

1. Why is GraphQL susceptible to the N+1 problem, and how does DataLoader solve it internally?

Every GraphQL field has its own resolver. When querying a list of users and their posts, the users resolver may execute one SQL query, but each user's posts resolver executes another query, leading to N additional database calls. DataLoader batches all pending requests during the same event loop tick, groups identical keys, performs a single batched query like SELECT * FROM posts WHERE user_id IN (...), and caches the result for the remainder of the request. This reduces N+1 queries into a constant number of database calls without changing resolver code.

2. How does GraphQL execute nested queries internally?

GraphQL first parses the incoming query into an Abstract Syntax Tree (AST), validates it against the schema, and creates an execution plan. It starts from the root Query resolver and recursively invokes child resolvers for every requested field. Parent resolvers return objects that become the source parameter for child resolvers. Since sibling resolvers are independent, many GraphQL implementations execute them concurrently, improving overall performance.

3. Why can GraphQL APIs become vulnerable to denial-of-service attacks?

Clients can construct deeply nested or highly expensive queries that recursively traverse relationships, forcing the server to execute thousands of resolver calls and database queries. Unlike REST where endpoints are predefined, GraphQL allows arbitrary query composition. Production systems therefore implement query depth limits, complexity scoring, persisted queries, authentication, and rate limiting to prevent abuse.

4. What are persisted queries, and why do large companies use them?

Instead of sending the complete GraphQL query each time, clients send only a hash. The server maps the hash to a previously stored query. This reduces request size, improves CDN caching, prevents arbitrary query execution, and blocks malicious clients from sending expensive ad-hoc queries. Facebook, Shopify, and Apollo commonly use persisted queries in production.

5. Explain GraphQL schema stitching versus federation.

Schema stitching merges multiple GraphQL schemas inside a gateway by manually combining resolvers. Federation allows independent teams to own separate subgraphs while a gateway automatically composes them into one unified schema. Federation scales significantly better for large organizations because each service evolves independently without requiring a centralized schema owner.

6. Why does gRPC achieve lower latency than REST?

gRPC uses HTTP/2 with persistent multiplexed connections, binary Protocol Buffer serialization, HPACK header compression, and generated code instead of reflection-heavy JSON parsing. Binary serialization is smaller and faster to encode/decode than JSON, while multiplexing eliminates head-of-line blocking caused by multiple HTTP/1.1 connections.

7. What happens internally when a gRPC client makes a request?

The client serializes the request object into Protocol Buffers, frames it using HTTP/2, and sends it over an existing TCP connection. The server receives the frame, deserializes the protobuf message, invokes the generated service handler, serializes the response, and streams it back over the same multiplexed connection. The connection usually remains open for future RPCs.

8. How does HTTP/2 multiplexing improve gRPC performance?

HTTP/2 allows multiple independent streams over a single TCP connection. Unlike HTTP/1.1 where requests often wait for previous ones or require multiple TCP connections, multiplexing lets many RPCs execute simultaneously. This reduces TCP handshakes, TLS negotiations, connection overhead, and latency while improving throughput.

9. What are gRPC interceptors, and where are they commonly used?

Interceptors are middleware that execute before and after RPC handlers. They are commonly used for authentication, authorization, distributed tracing, logging, metrics collection, retries, request validation, and panic recovery. They provide centralized cross-cutting functionality similar to middleware in REST frameworks.

10. Explain flow control in HTTP/2 and why it matters for gRPC streaming.

HTTP/2 implements flow control by assigning a window size to every stream and connection. The sender cannot transmit more data once the window is exhausted until the receiver acknowledges consumed bytes by sending WINDOW_UPDATE frames. This prevents fast producers from overwhelming slower consumers and avoids excessive memory usage during streaming.

11. Why is Protocol Buffers backward compatible?

Every field has a unique numeric tag. Unknown fields are ignored during deserialization, allowing newer messages to be consumed by older services. Developers should never reuse deleted field numbers because older clients may interpret reused tags as different fields, causing corrupted data.

12. Why isn't gRPC commonly exposed directly to browsers?

Standard browsers do not expose all HTTP/2 features required by native gRPC, such as low-level framing and trailers. gRPC-Web introduces a proxy that translates browser requests into native gRPC requests understood by backend services.

13. Why is REST generally easier to cache than GraphQL?

REST resources have stable URLs that map directly to cache entries, making HTTP caching via CDNs straightforward. GraphQL commonly uses a single endpoint where different request bodies represent different queries, making caching significantly more complex. Client-side normalized caches such as Apollo Cache are often required.

14. In a microservice architecture, when would you choose REST over gRPC?

REST is preferred for public APIs because it is human-readable, browser-friendly, cacheable, and widely supported across languages and tools. gRPC is typically used for internal service-to-service communication where performance, streaming, strict contracts, and low latency are higher priorities than interoperability.

15. Why do many companies expose GraphQL externally while using gRPC internally?

GraphQL provides frontend developers with flexible data fetching and minimizes client requests. Backend services communicate using gRPC because binary serialization, HTTP/2 multiplexing, and strongly typed contracts provide much higher efficiency. The GraphQL gateway simply orchestrates multiple gRPC calls before returning a unified response.

16. How would you optimize a GraphQL API serving millions of requests per day?

Use DataLoader batching, Redis caching, persisted queries, query complexity analysis, CDN caching for persisted queries, resolver-level metrics, database indexing, connection pooling, response compression, horizontal scaling, and distributed tracing to identify bottlenecks.

17. A gRPC streaming service starts consuming excessive memory. What could be the cause?

The producer may be sending messages faster than consumers can process them, leading to buffered messages in memory. Missing flow-control handling, large message sizes, insufficient backpressure, or slow downstream services can all contribute. Monitoring stream window sizes and limiting outstanding messages are common solutions.

18. Why is GraphQL introspection often disabled in production?

Introspection exposes the complete schema, including available queries, mutations, object relationships, and field documentation. Attackers can use this information to understand the API surface and craft expensive or malicious queries. Many production environments disable introspection for unauthenticated users while keeping it enabled for internal tooling.

19. How would you version APIs in GraphQL and gRPC?

GraphQL generally avoids explicit versioning by adding new fields and deprecating old ones while maintaining backward compatibility. gRPC versions APIs through protobuf evolution, package namespaces, or entirely new service definitions when breaking changes become unavoidable.

20. During a system design interview, how would you justify choosing GraphQL, REST, or gRPC?

The choice depends on consumers and requirements. GraphQL is ideal when clients require flexible data retrieval and multiple frontend applications consume the same backend. REST is preferable for public APIs requiring simplicity, caching, and broad ecosystem support. gRPC is the best choice for internal microservices that demand low latency, streaming, and high throughput. Large-scale systems like Netflix or Uber often combine all three, using GraphQL as an API gateway, REST for third-party integrations, and gRPC for internal service communication.

21. How does GraphQL resolve fields in parallel, and are there situations where resolvers execute sequentially?

GraphQL executes a query by traversing its Abstract Syntax Tree (AST). Starting from the root Query resolver, it invokes the resolver for each requested field. Once a parent resolver has returned an object, all of that object's child field resolvers become eligible for execution. Since sibling fields do not depend on one another, GraphQL implementations commonly execute them concurrently using asynchronous execution (Promises in Node.js, goroutines in Go, async tasks in Java, etc.).

However, parent-child relationships remain sequential because child resolvers require the parent object as input. For example, GraphQL cannot resolve user.posts.comments until the user resolver returns, followed by the posts resolver. Only after posts are available can comment resolvers begin.

This execution model means GraphQL achieves parallelism only where dependencies do not exist. Excessive nesting can therefore create long dependency chains even if sibling fields execute concurrently. Senior engineers often combine DataLoader, asynchronous database calls, caching, and resolver batching to maximize concurrency without overwhelming downstream services.

22. Why doesn't GraphQL automatically solve performance problems despite allowing clients to fetch exactly what they need?

GraphQL solves network efficiency, not backend efficiency. It prevents over-fetching and under-fetching by allowing clients to request only the required fields, but it does not dictate how those fields are retrieved internally.

If each resolver independently queries the database, a single GraphQL request may generate hundreds or even thousands of SQL queries. Deeply nested relationships amplify this problem, producing the classic N+1 issue.

Furthermore, GraphQL introduces additional CPU overhead for parsing, validation, authorization, query planning, resolver execution, and response serialization. Complex nested queries may consume far more CPU than equivalent REST endpoints.

Production GraphQL systems therefore require resolver optimization, batching, caching, database indexing, persisted queries, query complexity analysis, rate limiting, distributed tracing, and careful schema design. Without these optimizations, GraphQL can become slower than REST despite sending fewer bytes across the network.

23. How does gRPC streaming differ from repeatedly making unary RPC calls?

Unary RPC follows the traditional request-response model where each request creates an independent HTTP/2 stream. Although HTTP/2 keeps the TCP connection alive, every request still incurs metadata exchange, message framing, serialization, scheduling, and response processing.

Streaming establishes a long-lived logical communication channel. Once the stream is created, multiple messages can flow in one or both directions without reopening new RPCs. This significantly reduces protocol overhead while allowing messages to arrive immediately instead of waiting for complete request-response cycles.

Streaming is particularly useful for chat applications, live dashboards, financial market feeds, multiplayer games, telemetry pipelines, video analytics, IoT devices, and log aggregation systems.

However, streaming introduces additional complexity including flow control, connection management, heartbeats, retries, reconnection logic, message ordering, and backpressure handling. Long-lived streams also consume server resources, making proper lifecycle management essential.

24. Explain backpressure in gRPC streaming and why it is important.

Backpressure is the mechanism by which a slow consumer tells a fast producer to temporarily reduce or stop sending data. Without backpressure, producers can continuously generate messages until server memory becomes exhausted.

HTTP/2 implements flow control using window sizes at both the stream level and connection level. Each receiver advertises how many bytes it is currently willing to accept. As data is processed, WINDOW_UPDATE frames increase the available window, allowing additional data to be transmitted.

Imagine a telemetry service sending one million sensor updates per second while the downstream analytics service processes only one hundred thousand. Without flow control, buffered messages would accumulate indefinitely, eventually causing excessive memory consumption and service crashes.

Production systems therefore combine HTTP/2 flow control with bounded queues, worker pools, adaptive batching, rate limiting, and monitoring to maintain stable throughput even under heavy load.

25. How would you design a GraphQL API gateway sitting in front of hundreds of microservices?

The GraphQL gateway should not contain business logic. Instead, it acts as an orchestration layer responsible for authentication, authorization, query validation, query planning, response aggregation, caching, and request routing.

Each domain should own its own schema through GraphQL Federation or a similar architecture. The gateway composes these schemas into one unified API while delegating execution to the owning services.

Communication between the gateway and backend services is often performed using gRPC for efficiency. Frequently requested entities should be cached using Redis or in-memory caches, while DataLoader batches identical requests to downstream services.

Observability is equally important. Every GraphQL request should generate distributed traces that record resolver execution time, downstream service latency, cache hit ratios, database performance, and request complexity. This enables engineers to quickly identify bottlenecks when latency increases.

26. Why is HTTP/2 multiplexing not a complete solution to the Head-of-Line Blocking problem?

HTTP/2 eliminates application-level Head-of-Line Blocking by allowing multiple independent streams over a single TCP connection. Requests no longer wait for previous requests to complete before being transmitted.

However, HTTP/2 still relies on TCP. TCP guarantees ordered delivery, meaning if one packet is lost, subsequent packets cannot be delivered until the missing packet is retransmitted. Consequently, all HTTP/2 streams sharing that TCP connection temporarily stall.

HTTP/3 addresses this limitation by replacing TCP with QUIC, which operates over UDP. QUIC maintains independent streams that can continue progressing even when packets belonging to another stream are lost.

This distinction frequently appears in senior networking interviews because many engineers incorrectly assume HTTP/2 completely removes Head-of-Line Blocking.

27. How would you monitor and troubleshoot a slow GraphQL production API?

Monitoring begins at multiple layers. At the API level, collect request latency, throughput, error rates, query depth, query complexity, cache hit ratio, and resolver execution time.

Each resolver should emit metrics so that engineers can determine which field contributes the most latency. Distributed tracing tools such as OpenTelemetry allow requests to be traced across GraphQL, downstream services, databases, caches, and message queues.

Database performance should be analyzed using execution plans, slow query logs, connection pool statistics, and index utilization. Frequently requested objects should be cached, while expensive joins may need denormalization or precomputation.

Rather than treating GraphQL as a single API request, production teams view every resolver as an independent performance unit with its own latency and failure metrics.

28. Why do many organizations prefer gRPC for internal microservices but REST for external APIs?

Internal services are usually controlled by the same engineering organization. Since both client and server are known, they can safely share Protocol Buffer definitions, generated client libraries, and HTTP/2 infrastructure.

Binary serialization reduces bandwidth, generated code improves type safety, streaming enables real-time communication, and persistent connections reduce latency. These advantages become significant when thousands of services communicate continuously.

External APIs have different requirements. Third-party developers expect browser compatibility, human-readable JSON, simple debugging with curl or Postman, CDN caching, and broad language support. REST satisfies these expectations much better.

Consequently, many companies expose REST or GraphQL publicly while using gRPC internally for service-to-service communication.

29. Suppose one GraphQL query triggers requests to ten downstream microservices. How would you ensure reliability?

A GraphQL gateway should treat every downstream service as potentially unreliable. Each outbound request should have deadlines, timeouts, retries with exponential backoff (only for idempotent operations), circuit breakers, and bulkheads to prevent cascading failures.

Independent resolvers should execute concurrently wherever possible. Partial failures should return GraphQL's built-in partial response format, allowing successfully resolved fields to be returned while failed fields appear in the errors section.

Frequently requested reference data should be cached, while distributed tracing should capture every downstream dependency involved in fulfilling the query.

Designing GraphQL gateways therefore becomes less about GraphQL itself and more about applying distributed systems principles such as resilience, fault isolation, graceful degradation, and observability.

30. Compare GraphQL, REST, and gRPC from the perspective of scalability, maintainability, observability, and operational complexity.

REST offers the simplest operational model. Individual endpoints map naturally to business resources, integrate well with HTTP infrastructure, and leverage browser support and CDN caching. Its simplicity makes debugging and monitoring relatively straightforward.

GraphQL greatly improves frontend flexibility by allowing clients to request exactly the required data through a single endpoint. However, this flexibility shifts complexity to the server, requiring resolver optimization, schema governance, batching, query complexity analysis, caching strategies, and sophisticated observability. Large GraphQL deployments often require dedicated platform teams.

gRPC delivers the highest performance for service-to-service communication through Protocol Buffers, HTTP/2 multiplexing, streaming, and generated contracts. It excels in low-latency distributed systems but introduces operational challenges such as protobuf versioning, stream management, flow control, observability of binary traffic, and browser compatibility.

In modern large-scale architectures, these technologies are rarely competitors. Instead, they complement one another: GraphQL often serves as the client-facing API gateway, REST powers public integrations, and gRPC forms the high-performance communication layer between internal microservices.
