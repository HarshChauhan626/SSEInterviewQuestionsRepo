URL Shortener Suddenly Crashes During IPL Finals
CDNRedisHot KeysHorizontal Scaling
⌄
// scenario Your URL shortener normally handles traffic well. During IPL finals, a celebrity shares a shortened URL and the redirect service starts failing. Traffic goes from 1K req/s to 500K req/s in under a minute.
// interview answer

"Let me think through this systematically. The root cause is almost certainly a combination of a hot key problem in our cache, database saturation, and servers running out of capacity — all at once."

// architecture — before vs after fix
// root causes & fixes
  • 01
    Hot key in Redis: One single cache key (the celebrity's shortened URL) is being hit 500K times/sec. Redis becomes a bottleneck. Fix: Local in-memory cache (Guava/Caffeine) on each app server instance. If key is in local cache, we never hit Redis. Add Redis read replicas and route reads to them.
  • 02
    App server saturation: Single server can't handle 500K req/s. Fix: Auto-scaling group with horizontal scaling. Put a load balancer in front. Target CPU ~60%.
  • 03
    CDN for redirects: A URL redirect response is deterministic — the same short URL always maps to the same long URL. Push redirect responses to CDN edge nodes globally. 95%+ of traffic never reaches our servers.
  • 04
    Database bottleneck: If cache misses spike, DB gets hammered. Fix: Read replicas, connection pooling, and a write-through cache strategy.
💡 Key interview point: Always mention the tradeoff — local cache can serve stale data if the long URL is updated. For URL shorteners, that's acceptable since URLs rarely change. State this explicitly.

WhatsApp Users Receive Duplicate Messages
IdempotencyAt-Least-OnceDeduplication
⌄
// scenario Users are receiving the same WhatsApp message twice. This started after a recent deployment. Support tickets are pouring in.

"This is a classic distributed systems problem rooted in at-least-once delivery semantics. When a sender's client doesn't get an ACK, it retries — and if the server already processed the original message, the user gets a duplicate."

// message delivery flow with deduplication
// solution layers
  • 01
    Idempotency Key (msg_id): The sender generates a UUID per message before sending. On retry, it sends the same UUID. The server uses this UUID as a deduplication key.
  • 02
    Dedup Store: A Redis SET or a DB table stores processed msg_ids with a TTL (24–48h is enough since retries happen in seconds/minutes). Before processing, check if msg_id exists. If yes, return 200 OK but don't process again.
  • 03
    Client-side dedup: Receiver client also tracks recently seen msg_ids and suppresses rendering duplicates. Defense in depth.
  • 04
    ACK mechanism: Server sends ACK with sequence number. Client only retries if no ACK received within a timeout window.
// Pseudocode — server-side dedup function handleMessage(msg): if redis.SISMEMBER("seen_msgs", msg.id): return "OK" // Already processed, silent ack redis.SADD("seen_msgs", msg.id) redis.EXPIRE(msg.id, 86400) // 24h TTL deliverToRecipient(msg) return "OK"
⚠️ Exactly-once delivery is impossible in distributed systems — you can only get at-least-once or at-most-once. The dedup layer gives us the effect of exactly-once from the user's perspective.

Swiggy Shows Wrong Rider Location
WebSocketsGPS PollingEventual Consistency
⌄
// scenario The delivery partner is 3 km away according to GPS, but the customer app shows him standing at their door. There's a location lag of several minutes.

"This is a real-time data freshness problem. Location data needs to flow from the rider's device to our servers to the customer's screen in near real-time, but something in that pipeline is stale or broken."

// real-time location pipeline
  • 01
    GPS polling interval: Rider app should push location every 3–5 seconds (when in motion). If battery saver kicks in, interval becomes 30s. Fix: Adaptive polling — frequent when speed > 0, slower when stationary.
  • 02
    WebSocket over polling: Customer app should use WebSocket connection to receive location pushes, not HTTP polling. Polling every 5s means up to 5s lag + network latency. WebSocket = sub-second push latency.
  • 03
    Redis GEO commands: Store current rider positions using Redis GEOADD. O(log N) inserts, fast nearest-neighbor queries. Pub/Sub for pushing updates to WebSocket servers.
  • 04
    Eventual consistency acknowledgement: Show "last updated X seconds ago" in UI. If no update in 15s, show "Location unavailable" rather than stale data.
ℹ️ The old location being shown at the destination could be a stale cache issue — if the system cached the last known position and that cache wasn't invalidated when the order was delivered. Always set TTLs on location data.

BookMyShow Accidentally Sells the Same Seat Twice
Race ConditionsDistributed LockingOptimistic vs Pessimistic
⌄
// scenario Two users — User A and User B — both select seat 12C in a Friday night show. Both get confirmation emails. Both show up at the theater.

"This is a race condition — a classic concurrency bug. Two transactions read the seat as 'available', both proceed to book, both succeed. The read and the write weren't atomic. Let me walk through the solutions."

// the race condition visualized
-- BROKEN: Both transactions read AVAILABLE at the same time T1: SELECT status FROM seats WHERE id='12C' → 'AVAILABLE' ✓ T2: SELECT status FROM seats WHERE id='12C' → 'AVAILABLE' ✓ (race!) T1: UPDATE seats SET status='BOOKED', user='A' WHERE id='12C' ← overwrites T2: UPDATE seats SET status='BOOKED', user='B' WHERE id='12C' ← both succeed!
// solution 1 — pessimistic locking
-- SELECT FOR UPDATE acquires a row-level lock BEGIN; SELECT * FROM seats WHERE id = '12C' AND status = 'AVAILABLE' FOR UPDATE; -- T2 will BLOCK here until T1 commits/rollbacks -- If found: book it UPDATE seats SET status='BOOKED', user='A' WHERE id='12C'; COMMIT;
// solution 2 — optimistic locking
-- Add a version column, check it on update SELECT id, status, version FROM seats WHERE id='12C'; -- Got: { status: 'AVAILABLE', version: 7 } UPDATE seats SET status='BOOKED', version=8 WHERE id='12C' AND version=7; -- Only succeeds if nobody changed it -- If rows_affected = 0, someone else grabbed it → retry/fail
// solution 3 — distributed lock for temporary hold
  • 01
    User selects seat → acquire Redis lock: SET seat:12C:lock userA NX PX 300000 (5 min TTL)
  • 02
    While locked: seat shows as "reserved" to all other users
  • 03
    User completes payment → DB write → release lock
  • 04
    If payment fails or times out → lock auto-expires → seat becomes available again
ApproachProCon
Pessimistic LockSimple, strong guaranteeDB lock contention at scale
Optimistic LockHigh concurrency, no blockingRetry storms under high contention
Redis Distributed LockScales, user experience (holds seat)Lock expiry edge cases, Redis SPOF
💡At BookMyShow scale: use a Redis temporary hold for UX (show timer to user), then an optimistic lock on final DB write. Belt and suspenders.

Netflix Starts Buffering Right After a New Season Release
CDNCache PrewarmingAuto-scaling
⌄
// scenario Stranger Things Season 5 drops. 20 million users hit play in the first hour. CDN cache is cold. Origin servers are getting slammed.

"Netflix actually pre-warms their CDN before a major release. The fact that we're buffering means we either skipped that, had a cache miss storm, or underestimated the auto-scaling lead time. Let me break down the full solution."

// netflix-style video delivery architecture
  • 01
    CDN Pre-warming: Before release, Netflix pushes video chunks (especially the first 2 minutes) to all CDN edge nodes globally. When users hit play, the CDN serves it — origin never gets touched. This is done via "cache seeding" scripts.
  • 02
    Adaptive Bitrate (ABR): Serve multiple quality levels (240p → 4K). Client player starts at lower quality immediately and upgrades as bandwidth allows. User perceives no buffer.
  • 03
    Auto-scaling with lead time: Set CloudWatch alarms to trigger scale-out 15 minutes before scheduled release. Don't wait for CPU to spike — you'll be too late.
  • 04
    Regional traffic shaping: If one region's CDN is overwhelmed, route overflow to the next-closest region. Geo-DNS based routing.
💡Netflix Open Connect — their custom CDN — places physical servers inside ISPs. When you stream, you're often hitting a box in your ISP's data center, not a cloud server. That's how they handle 500Gbps+ of traffic.

Uber Surge Pricing During Rain
Stream ProcessingKafkaReal-time Analytics
⌄
// scenario It starts raining in Mumbai. Within seconds, ride requests spike 10x. Uber needs to compute surge pricing dynamically and reflect it in the app instantly.

"Surge pricing is a real-time supply-demand matching problem. We need a streaming pipeline that continuously computes demand vs. supply per geo-cell and emits a multiplier. Latency needs to be sub-second."

// surge pricing pipeline
  • 01
    H3 Geo-indexing: Divide the city into hexagonal cells (Uber's H3 library). Each cell tracks its own demand/supply independently. Mumbai cell 8f2830... sees 50 requests, 3 available drivers → surge.
  • 02
    Kafka as event backbone: Every ride request and every driver location update is a Kafka event. Topics: ride.requests and driver.locations.
  • 03
    Flink sliding window: 5-minute sliding window, compute demand/supply ratio per cell. Ratio > 2 → 1.5x surge. Ratio > 5 → 2.5x surge. Emit to Redis.
  • 04
    App reads surge from Redis: Sub-millisecond reads. Price shown in app reflects current surge. Updates every 30s to user.
ℹ️Also factor in supply signals: dispatching idle drivers in nearby cells, incentivizing drivers with bonuses (a business logic layer on top of the compute pipeline).

Amazon Cart Shows Old Data
Cache InvalidationEventual ConsistencyEvent-Driven
⌄
// scenario A user removes shoes from their cart on mobile. Refreshes on laptop — shoes still there. 3 minutes later, laptop finally shows the correct cart.

"This is the cache invalidation problem — infamously one of the hardest problems in CS. The mobile client removed the item from the DB, but the laptop is reading from a stale cache. Let me walk through how to fix this correctly."

  • 01
    Write-through cache: When item is removed via mobile → write to DB AND invalidate (delete) the cart cache key simultaneously in the same transaction. Next read will be a cache miss → fetch fresh from DB → repopulate cache.
  • 02
    Event-driven invalidation: Cart update event published to Kafka. Cache invalidation service consumes the event and deletes the stale key from Redis across all regions.
  • 03
    Cache key per session vs per user: If each device has its own session cache, invalidating one doesn't help the other. Use user-level cache keys: cart:userId:12345 — same key for all devices.
  • 04
    Short TTL: Set TTL of 60–120s on cart cache. Even without explicit invalidation, stale data self-corrects within 2 minutes. For cart, that's acceptable.
  • 05
    Version/ETag in response: Return a cart version number. Client sends current version in next request. If server version is newer, return fresh data; otherwise 304 Not Modified.
⚠️Phil Karlton: "There are only two hard things in Computer Science: cache invalidation and naming things." — Amazon cart is the canonical example. Mention trade-offs: strict consistency (always fresh, slower) vs eventual consistency (possibly stale, faster).

Instagram Notification Storm
Fan-outAsync ProcessingBackpressure
⌄
// scenario Virat Kohli uploads a post. He has 260 million followers. All of them should get a push notification "Virat posted something." That's 260M notifications to deliver instantly.

"This is a classic fan-out on write vs read problem. 260 million notifications cannot be sent synchronously. We need an async, queued, batched, prioritized delivery pipeline."

// fanout architecture — celebrity post
  • 01
    Async fan-out: Post event → Fan-out Service reads follower list in batches of 10K → enqueues notification jobs. Don't try to do 260M writes synchronously.
  • 02
    Queue sharding: Partition notification queues by user_id mod N. 100 queues → each handles 2.6M users. Workers process in parallel.
  • 03
    Priority tiers: Active users in the last hour → high priority queue (deliver in <30s). Inactive users → low priority queue (deliver over minutes). This improves perceived performance.
  • 04
    Backpressure: If APNs rate limits us, use exponential backoff and put items back in queue. Don't drop — but also don't storm the downstream service.
  • 05
    For mega-celebrities: Consider fan-out on read — don't pre-compute individual notifications. Instead, when the user opens the app, check if they follow anyone who posted recently. Less write load, slightly higher read latency.

Payment Deducted But Order Failed
Saga PatternDistributed TransactionsCompensation
⌄
// scenario User checks out on Zomato. Payment goes through (₹450 deducted), but the order creation service crashes immediately after. User has no order, but money is gone.

"This is the most important distributed systems problem in e-commerce. We can't use a traditional ACID transaction across two microservices — payment service and order service run different databases. The solution is the Saga pattern."

// saga pattern — choreography style
// idempotency key — the critical detail
// Payment request with idempotency key { "amount": 450, "user_id": "u_123", "idempotency_key": "checkout_session_abc_v1" } // If we retry, same key → same result, no double charge // Payment service stores: key → result for 24h
  • 01
    Saga choreography: Each service publishes events. Next service listens and acts. On failure, publish a failure event → previous services run compensation actions (refund, un-reserve, etc.)
  • 02
    Idempotency keys: Every payment request includes a unique key (order ID + attempt number). Payment service stores results for 24h. Retries return the cached result instead of charging again.
  • 03
    Outbox pattern: Payment service writes to its DB and to an outbox table in the same DB transaction. A separate process reads the outbox and publishes to Kafka. Guarantees the event is never lost even if the process crashes mid-way.
🚨This is the most asked distributed systems question in senior backend interviews. Know the Saga pattern, know the Outbox pattern, and always mention idempotency keys.

YouTube Video Processing Pipeline
Async WorkersEncoding PipelineChunk Processing
⌄
// scenario A user uploads a 2GB 4K video. Within minutes, they expect it to stream in 360p, 720p, 1080p, and 4K with chapters, thumbnails, and captions.

"This is a classic async processing pipeline problem. Video processing is CPU-intensive, time-consuming, and can fail at any stage. We must never do this synchronously in a web request."

// video processing pipeline
  • 01
    Chunked upload: Client splits video into 5MB chunks, uploads in parallel with resumability. Reassembled in S3 using multipart upload. Never fail a 2GB upload on a 1.9GB progress.
  • 02
    Parallel encoding workers: Multiple quality levels encoded in parallel across separate worker machines. 360p doesn't wait for 4K to finish — 360p is ready first, user can start watching.
  • 03
    Progressive availability: DB record for video shows status: PROCESSING. As each quality becomes available, update: qualities_ready: [360p, 720p]. Video is viewable as soon as 360p is done.
  • 04
    Idempotent workers: If a worker crashes mid-encode, it restarts from beginning. To avoid re-encoding from scratch, use segment-based processing — encode 10s segments independently.

Kafka Consumer Processes Same Event Twice
Kafka OffsetsExactly-OnceIdempotent Consumers
⌄
// scenario Your notification consumer processed a Kafka event and sent the notification — but crashed before committing the offset. On restart, it re-reads the same event and sends duplicate notification.

"Kafka provides at-least-once delivery by default. If a consumer crashes after processing but before committing the offset, it will reprocess on restart. This is expected behavior — we need to design consumers to be idempotent."

// THE PROBLEM — crash between process and commit consumer.poll() → gets event E1 at offset 42 process(E1) → sends notification ✓ consumer.crash() → offset 42 never committed! // On restart: consumer.poll() → gets E1 AGAIN at offset 42 process(E1) → sends notification AGAIN 💥 // THE FIX — idempotent processing function process(event): if db.exists(`processed:${event.id}`): return // Already done, skip safely begin transaction: sendNotification(event) db.insert(`processed:${event.id}`) commit
  • 01
    Dedup table: Store processed event IDs in a DB table with a unique constraint. Before processing, check if ID exists. If yes, skip and commit offset.
  • 02
    Kafka's exactly-once semantics (EOS): Use enable.idempotence=true + transactional producer. Works for Kafka-to-Kafka flows. For Kafka-to-DB, you still need the dedup approach above.
  • 03
    Manual offset commit: Only commit offset after successfully processing AND recording in DB. Use enable.auto.commit=false.
  • 04
    Dead Letter Queue (DLQ): If an event fails after N retries, move it to a DLQ topic. Don't block the main consumer. Alert on DLQ size.

Order Events Arrive Out of Sequence
Event OrderingKafka PartitionsSequence Numbers
⌄
// scenario Your order tracking service receives "Order Delivered" before it receives "Order Shipped." The UI shows delivered but then flips back to shipped when the late event arrives.

"Event ordering is a fundamental distributed systems challenge. In Kafka, ordering is only guaranteed within a single partition. If order events land on different partitions, they can arrive out of order. Let me explain why and how to fix it."

  • 01
    Partition by order_id: Use order_id as the Kafka message key. All events for the same order will always go to the same partition (consistent hashing). Same partition → strict ordering.
  • 02
    Sequence numbers: Each event carries a sequence_num. Consumer applies events only in order: if current state is sequence 3, and you receive sequence 5, hold event 5 in a buffer until sequence 4 arrives.
  • 03
    State machine validation: Define valid state transitions: PLACED→CONFIRMED→SHIPPED→DELIVERED. If DELIVERED arrives when current state is PLACED, reject it as invalid. Emit alert for investigation.
  • 04
    Timestamp-based resolution: If two conflicting events arrive, take the one with the later event_timestamp, not the one that arrived later.
// State machine — reject invalid transitions VALID_TRANSITIONS = { "PLACED": ["CONFIRMED", "CANCELLED"], "CONFIRMED": ["SHIPPED", "CANCELLED"], "SHIPPED": ["DELIVERED", "RETURNED"], "DELIVERED": ["RETURNED"] } if newState not in VALID_TRANSITIONS[currentState]: publishToDLQ(event) // Out-of-order — handle separately

Notification Service Crashes During Flash Sale
BackpressureRate LimitingLoad Shedding
⌄
// scenario Big Billion Day begins. 5 million "sale started!" notifications need to go out. The notification service starts timing out. The queue grows. Eventually, OOM crash.

"The notification service is a bounded resource being asked to do unbounded work. The solution is backpressure — slow down the producer, protect the consumer, and shed load gracefully."

  • 01
    Queue buffering: Never send directly — always go through a durable queue (SQS, Kafka). The queue absorbs burst traffic. Workers pull at a sustainable rate. If APNs allows 1000 req/s, consume from queue at exactly 1000/s.
  • 02
    Rate limiting at worker level: Token bucket per notification channel (APNs, FCM). Workers wait for tokens before sending. Smooth, predictable throughput.
  • 03
    Load shedding: If queue depth > 10M, prioritize active users. Drop notifications for users inactive > 30 days. They'll see it next time they open the app. Communicate tradeoff explicitly.
  • 04
    Horizontal scaling: Spin up more consumer workers (EC2 or Lambda). Lambda scales to thousands of concurrent invocations, each reading from SQS.
  • 05
    Pre-warm before sale: Scale up 30 minutes before the sale start time. Auto-scaling reacts to load; pre-warming anticipates it.
💡Backpressure rule: the system should push back against producers as fast as the consumers can consume — never let work pile up unboundedly in memory. The queue is your shock absorber.

Why Instagram Avoids JOINs at Scale
DenormalizationNoSQLRead Optimization
⌄
// scenario Why do companies like Instagram, Twitter, and Facebook avoid SQL JOINs in their main production queries, even though they use relational databases?

"At Instagram's scale — 2 billion users — a JOIN that looks like O(n) in dev becomes O(n × m) across sharded databases on different machines. You can't JOIN across database servers. The answer is denormalization and data locality."

Normalized (JOIN)Denormalized (no JOIN)
User table + Post table → JOIN on user_idPost row stores: post_id, content, user_id, username, user_avatar
Single source of truthData duplicated across tables
Update username once, everywhere consistentUpdate username → update millions of post rows
Good for low-volume OLTPGood for high-volume read-heavy social feeds
  • 01
    Sharding kills JOINs: Instagram shards user data across ~1000 PostgreSQL shards. To JOIN users with posts, you'd need cross-shard queries — impossible efficiently. Instead, embed user data in post documents.
  • 02
    Read:write ratio: Instagram has 10,000:1 read-to-write ratio. Optimize for reads by denormalizing. Pay the cost at write time, not read time.
  • 03
    Application-level joins: Fetch post IDs from one service, fetch user data from user service, merge in application layer. Cache user data aggressively (user profiles rarely change).
  • 04
    Precomputed feeds: Don't compute the home feed on read. When someone posts, fan out to followers' precomputed feed lists (stored in Redis). Read is a single list fetch — no joins at all.

DynamoDB Hot Partition Problem
Partition KeysShardingHot Partitions
⌄
// scenario Your DynamoDB table uses celebrity user_id as the partition key for a fan activity table. When a celebrity does a live session, all 50M fans write activity records simultaneously → one partition explodes, ProvisionedThroughputExceededException everywhere.

"DynamoDB distributes data across partitions based on the partition key. If one key receives disproportionate traffic, that physical partition becomes a bottleneck. The solution is to artificially distribute the hot key across multiple partitions."

// BAD: All activity for celebrity_123 goes to one partition PK: "celebrity_123" // GOOD: Add random suffix to spread across 10 partitions suffix = random.randint(0, 9) PK: f"celebrity_123_{suffix}" // → celebrity_123_0, celebrity_123_1 ... celebrity_123_9 // 10x write throughput immediately // To READ all activity: scatter-gather results = [] for suffix in range(0, 10): results += query(PK=f"celebrity_123_{suffix}") merge_and_sort(results)
  • 01
    Write sharding: Append random suffix 0–N to partition key. Writes spread across N partitions. N = expected peak writes / partition throughput limit.
  • 02
    Scatter-gather reads: To read all data for the entity, query all N partition variants in parallel and merge. Adds latency but enables scale.
  • 03
    DynamoDB DAX: Add a DAX (in-memory) cache layer in front of DynamoDB. Read-heavy workloads hit DAX (~microseconds) instead of DynamoDB (~milliseconds).
  • 04
    On-demand capacity: Switch to on-demand pricing during celebrity events. AWS auto-scales instantly. More expensive per-RCU but avoids throughput exceptions during unpredictable spikes.

Redis Cache Causes Production Outage
Cache StampedeTTL JitterRequest Coalescing
⌄
// scenario At 3 AM, your Redis cache has a scheduled TTL expiration for 10 million keys (all set at the same time during a data load 24h ago). All expire simultaneously. Every request becomes a cache miss. Database gets 10 million queries at once. Database dies.

"This is called a Cache Stampede or Thundering Herd problem. Millions of simultaneous cache misses create a spike of database reads that the DB can't handle. The root cause was synchronized TTL expiration."

// BAD: All keys expire at exactly the same time redis.SET("product:1", data, EX=86400) // 24h redis.SET("product:2", data, EX=86400) // 24h // 10M keys, all expire at same instant 💥 // GOOD: TTL jitter — spread expirations base_ttl = 86400 jitter = random.randint(0, 3600) // ±1h random redis.SET(key, data, EX=base_ttl + jitter) // Expirations now spread over 1 hour, not 1 second
  • 01
    TTL Jitter: Add random offset to every TTL. If base TTL is 24h, set each key to 24h + random(0, 1h). Expirations spread across an hour instead of happening simultaneously.
  • 02
    Probabilistic early expiration: Before a key expires, proactively refresh it with a small probability. Key won't expire if it's refreshed before TTL hits.
  • 03
    Request coalescing (mutex lock): When a cache miss occurs, only ONE request goes to the DB. Others wait. Use Redis SETNX to acquire a "fetching" lock. When data is fetched, release lock and populate cache.
  • 04
    Multi-level caching: L1 = in-process memory (Caffeine/Guava), L2 = Redis, L3 = DB. Even if Redis is cold, L1 handles many requests. Stagger the warmup.
  • 05
    Circuit breaker on DB: If DB error rate > threshold, open circuit breaker, serve stale data or graceful degraded response. Better to serve slightly stale than return errors.

AI Chatbot Gives Wrong Answers (Hallucinations)
RAGVector DBPrompt Engineering
⌄
// scenario Your customer service AI chatbot confidently tells users that a product has features it doesn't have, or quotes prices that are wrong, or makes up return policies.

"This is the hallucination problem. LLMs generate plausible-sounding text based on patterns, but they don't know your company's internal data. The solution is Retrieval-Augmented Generation (RAG) — ground the LLM's answers in your actual data."

// RAG architecture
  • 01
    RAG pipeline: Embed all your product docs, FAQs, policies into a vector DB. When user asks a question, embed it, find top-5 similar documents, inject them into the prompt as context. LLM answers based on your data, not its training.
  • 02
    Prompt guardrails: Instruct the LLM: "Answer only based on the provided context. If you don't know, say 'I don't have information on this, please contact support.'" This is critical.
  • 03
    Confidence thresholds: If retrieved documents have low similarity score (<0.7), don't answer with AI — fall back to human agent or FAQ links.
  • 04
    Answer verification layer: Post-generate, verify key facts (price, product name) against source of truth. If mismatch → don't show AI answer, show fact-checked answer.

AI Platform Suddenly Becomes Very Expensive
Prompt OptimizationModel RoutingCaching Responses
⌄
// scenario You built an AI feature using GPT-4. It's wildly popular. Your monthly LLM bill goes from $2,000 to $80,000 in 3 weeks. You need to cut costs without degrading quality.

"LLM costs scale with tokens in + tokens out. The levers are: reduce tokens, cache results, route to cheaper models. Let me walk through each."

  • 01
    Semantic response caching: If 1000 users all ask "what are your business hours?" — that's 1000 identical LLM calls. Cache the response. Use the embedded query as the cache key. Exact match → instant cache hit. Semantic similarity > 0.95 → cache hit. Can eliminate 60–70% of LLM calls.
  • 02
    Model routing (cheap vs expensive): Simple questions → GPT-3.5 or Haiku (~10x cheaper). Complex reasoning → GPT-4. Use a classifier to route. You pay for GPT-4 only when you need it.
  • 03
    Prompt compression: Strip system prompts of redundancy. Use abbreviations for repeated terms. Remove whitespace. Every saved token = saved money. Use tools like LLMLingua.
  • 04
    Context window management: Don't send 50 messages of chat history for every turn. Summarize conversation history every 10 turns. Send summary + last 3 messages instead of full history.
  • 05
    Fine-tuned smaller model: Fine-tune GPT-3.5 or Mistral-7B on your specific domain. Domain-specific fine-tuned small model often beats generic GPT-4 at your task, at 1/20th the cost.
💡The 80/20 rule: usually 80% of your LLM traffic is simple, repetitive queries. Cache those. Spend your GPT-4 budget on the 20% of genuinely complex queries.

AI Search Feels Slow
Vector IndexingANN SearchRecall vs Latency
⌄
// scenario Your semantic search feature (find similar products using embeddings) takes 3–4 seconds per query. Users expect sub-200ms results.

"Exact nearest-neighbor search in high-dimensional space (768 dims) across 10M vectors is O(n × d) — impossibly slow at scale. The solution is Approximate Nearest Neighbor (ANN) search — accept tiny accuracy loss for massive speed gains."

AlgorithmSpeedRecallBest For
Exact KNNSlow (O(n))100%Small datasets <100K
HNSWFast (<10ms)~99%General purpose, large scale
IVF-PQVery Fast~95%Billion-scale, memory constrained
LSHFast~85%Simple, low-maintenance
  • 01
    HNSW index (Hierarchical Navigable Small World): Build a multi-layer graph structure. Query traverses the graph in O(log n) instead of O(n). Qdrant, Pinecone, Weaviate all use HNSW. Goes from 3s to <10ms at 10M vectors.
  • 02
    Pre-filtering: First apply metadata filters (category, price range) to reduce candidate set from 10M → 50K, then run ANN on smaller set. Much faster.
  • 03

    Expand-Contract (parallel-change pattern): Phase 1 adds the new column alongside the old (backward compatible). Phase 2 migrates all writes to the new column while reads support both. Phase 3 removes the old column after all instances are updated. Three separate deploys, zero downtime.
  • 02
    Online schema migration tools: ALTER TABLE on a large table locks the table for minutes (MySQL) or is safe but slow (PostgreSQL). Use pt-online-schema-change (Percona) or gh-ost (GitHub) — they create a shadow table, copy data in batches, and swap atomically. Zero lock on production table.
  • 03
    Blue-Green deployment: Maintain two identical environments (Blue = live, Green = new version). After testing Green, switch DNS from Blue to Green. Rollback = switch DNS back to Blue. Works for code deploys; requires careful handling for schema changes (Green's schema must be backward compatible with Blue traffic during cutover).
  • 04
    Feature flags for schema-tied features: If a new feature requires the new column, keep it behind a feature flag. Roll out the schema, then gradually enable the feature flag. Decouple schema migration timing from feature availability.

Autocomplete Search Needs to Handle 1 Million RPS
TrieRedis ZSETPrefix CacheTop-K
⌄
// scenario Your search bar sends a request on every keystroke. With 100K concurrent users typing at ~2 keystrokes/second, that's 200K RPS just for autocomplete. The naïve SQL LIKE 'query%' with a full-text index can't keep up — P99 latency is 800ms.

"Autocomplete is a prefix search problem on a Top-K query set. Precompute the top suggestions for every prefix and serve from in-memory structures. LIKE queries on large tables can never achieve <20ms at 200K RPS."

// Redis ZSET autocomplete — top suggestions per prefix // On query popularity update (after a successful search) for prefix in all_prefixes("iphone"): // "i","ip","iph","ipho","iphon","iphone" redis.ZINCRBY(f"auto:{prefix}", 1, "iphone") // On user keystroke (query time) function autocomplete(prefix): return redis.ZREVRANGE(f"auto:{prefix}", 0, 9) // top 10 // O(log N) per request. 1M RPS = 1M Redis ops/sec. Cluster handles it.
  • 01
    Precomputed prefix sets in Redis ZSET: For every popular search term, write the term to all of its prefix sets scored by search frequency. "iphone" goes into sets "i", "ip", "iph", etc. Autocomplete = single ZREVRANGE on auto:{prefix}. Sub-millisecond reads.
  • 02
    Limit prefix fan-out: A term with 20 characters has 20 prefixes. Writing to 20 Redis keys per popular query is fine. But with millions of tail queries, the number of keys explodes. Set a minimum prefix length of 2 characters and only maintain Top-1000 queries per prefix — prune the long tail.
  • 03
    Trie for exact prefix matching (alternative): An in-memory trie data structure enables O(prefix_length) lookup. For a service with <10M unique terms, fit the entire trie in RAM (one app server, 8GB). For larger vocab, distribute by prefix hash — requests for "i*" go to shard 1, "a*" go to shard 2.
  • 04
    Client-side debouncing: Instead of firing on every keystroke, fire 150-300ms after the last keystroke. Reduces RPS by 60-70% for typical typing speeds with zero user-visible quality degradation. The easiest optimization that requires no backend changes.
💡Google's search-as-you-type uses a combination: a trie for prefix matching, a large language model for fuzzy matching and intent detection, and a huge CDN cache for the most popular prefix queries. But for most products, Redis ZSET + client debouncing covers 99% of the use case.

Real-Time Fraud Detection Must Decide in 50ms
Stream ProcessingFeature StoreML InferenceRisk Scoring
⌄
// scenario A payment is submitted. You have 50ms to decide: approve or flag for review. The fraud model needs 40+ features: user transaction history, device fingerprint, location, merchant category, spending velocity in last 1 hour. The DB queries to compute these take 800ms.

"The 800ms query time means you're computing features at inference time from cold storage. The fix is a Feature Store — precompute and maintain features in low-latency storage, so inference time is just a lookup."

// fraud detection pipeline: feature store + ML inference
  • 01
    Feature Store with precomputed features: Maintain a Redis hash per user: fraud_features:{user_id} → {tx_count_1h: 3, tx_amount_1h: 450, unique_merchants_24h: 7, ...}. A Flink streaming job continuously updates these features as transactions arrive. At inference time, one Redis HGETALL = all 40 features in under 5ms.
  • 02
    Sliding window velocity features: "Transactions in the last 1 hour" is expensive to compute on demand (requires scanning all txns). Precompute with a Flink sliding window: on every transaction, update ZADD tx_history:{user_id} timestamp tx_id, then count with ZCOUNT in the last 3600s. Maintained incrementally — O(log n) per update.
  • 03
    Lightweight ML model for inference latency: Gradient Boosted Trees (XGBoost/LightGBM) infer in <5ms with 40 features — far faster than neural networks. Neural networks excel at raw data (images, text); tabular fraud features are the ideal case for gradient boosting. Don't reach for deep learning when GBT is sufficient.
  • 04
    Risk tier routing: Low risk (<0.1) → auto-approve. Medium risk (0.1-0.5) → approve + async review. High risk (0.5-0.8) → SMS/OTP verification step. Very high (>0.8) → block + notify user. This handles edge cases without binary approve/block creating too many false positives.

Log Pipeline Is Drowning in 10TB Per Day
SamplingLog AggregationStructured LoggingTiered Storage
⌄
// scenario Your platform generates 10TB of logs per day. Elasticsearch ingestion is falling behind. Log search latency is 45 seconds. Storage bill is $50K/month. Engineers stopped looking at logs because they're too slow and too expensive.

"Logging everything at DEBUG level without a retention strategy is a common mistake. The fix is log tiering, structured logging, and intelligent sampling — not more Elasticsearch capacity."

  • 01
    Log level discipline: DEBUG logs should never reach production Elasticsearch. Route DEBUG to a short-lived local buffer (30 minutes). INFO logs go to Elasticsearch with 7-day retention. WARN/ERROR go to Elasticsearch with 30-day retention and PagerDuty integration. This single change typically reduces log volume by 80%.
  • 02
    Structured logging (JSON): Replace unstructured messages like "Payment processed for user 123 in 45ms" with JSON: {"event": "payment_processed", "user_id": 123, "duration_ms": 45}. Structured logs are indexable, filterable, and parseable. You can search user_id:123 instead of regex-parsing freetext.
  • 03
    Tiered storage: Hot tier (Elasticsearch, <7 days) for fast search. Warm tier (S3 + Athena, 7-90 days) for ad-hoc SQL queries at $5/TB scanned. Cold tier (S3 Glacier, >90 days) for compliance at $0.004/GB/month. 90% of log queries are about the last 3 days — match your storage cost to your access patterns.
  • 04
    Tail-based sampling for traces: Don't trace every request. Sample 1% of successful requests, 100% of errors, 100% of slow requests (>1s). Tail-based sampling (Jaeger, OpenTelemetry) makes the sampling decision after the full trace is collected — you always capture errors even if they started as "normal" requests.
ℹ️Alternatives to Elasticsearch: ClickHouse handles 10TB/day at a fraction of the cost and is 10-100x faster for aggregation queries over large time ranges. Loki (Grafana) stores log metadata in Prometheus and raw text in S3 — much cheaper than indexing everything in Elasticsearch. Evaluate these before scaling your ES cluster.
